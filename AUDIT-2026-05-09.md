### ❌ FIXED TONIGHT

**Critical: `/account/reauth` inverted rate limit**
Line 915 had `if (checkAuthRateLimit(clientIp))` — opposite of every other auth endpoint. Effect: legitimate users hit 429 on their first 5 reauth attempts; brute-force was possible after the limit was "exhausted." Fixed to `if (!checkAuthRateLimit(clientIp))`. Server restarted and confirmed running.

### ⚠️ FLAGS (for Scott)

**1. Node process runs as root on Hetzner.**
The `citeback-ai.service` has no `User=` directive, so Node runs as root. If the Node process is compromised, the attacker has full root access. Best practice: create a dedicated `citeback` user with minimal permissions, change ownership of `/opt/citeback-ai/` to that user, and add `User=citeback` to the systemd service.

Steps to fix when ready:
```bash
useradd -r -s /bin/false citeback
chown -R citeback:citeback /opt/citeback-ai/
# Edit /etc/systemd/system/citeback-ai.service: add User=citeback
systemctl daemon-reload && systemctl restart citeback-ai
```

**2. `setCorsHeaders` function in server.js is dead code.**
The function is defined but never called (Caddy handles all CORS headers). Minor but confusing. Can be removed in a cleanup pass.

**3. Old admin secret in git history.**
`docs/internal/PHASE1-BUILD.md` previously contained `45f54c8f4718a97c9afe65c4c935bc64d20a0df5b88a0659`. The secret is redacted in the latest commit, but the old value remains in git history. Since the secret is already rotated (Hetzner uses `3ba95ce...`), the risk is low — anyone who tries it will get 401. But if you ever want a truly clean history, a `git filter-repo` run can purge it. Low priority.
