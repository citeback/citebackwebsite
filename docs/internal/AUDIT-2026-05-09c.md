### ⚠️ ARCHITECTURAL FLAGS (no immediate fix — design decisions)

**1. Reauth window is client-side only — 🟡 Medium**

`/account/reauth` returns `reauthUntil = Date.now() + 5 * 60 * 1000`. The client stores this in `reauthUntilRef.current` (a React ref, not persisted). The server never checks if a caller has recently reauthenticated. Only `ProposeModal.jsx` uses `isReauthed()` — and it's a UI gate, not a server security control.

**Current risk:** Low. `ProposeModal` submits to `/api/propose` which has no financial action (just posts to proposals.jsonl). No sensitive endpoint actually requires server-side reauth enforcement today.

**Future risk:** High, if endpoints like campaign wallet update or email change are later guarded by reauth client-side but not server-side.

**Recommendation:** Before adding any high-stakes wallet/financial endpoint, add server-side reauth enforcement: store `reauthUntil` in the session (or a short-TTL in-memory map keyed by userId), and check it on sensitive routes.

**2. No session invalidation on password reset — 🟡 Medium**

After a successful password reset, existing JWTs remain valid until their 7-day expiry. An attacker who stole a session cookie retains access for up to 7 days even after the user resets their password.

This is a fundamental JWT limitation. Stateless JWTs have no revocation mechanism by design.

**Options (in order of increasing complexity):**
- Short-term: reduce JWT expiry from 7d → 24h (tradeoff: users re-login more)
- Medium-term: add a `password_version` counter to users table; include in JWT; reject tokens where JWT version < current version
- Long-term: token blocklist (Redis or SQLite) for O(1) revocation

**Current risk:** Low pre-launch (no real users, no adversaries). Medium post-launch.

## Section 2: Rate Limiting — X-Forwarded-For Analysis

### Finding: XFF Bypass — Not Exploitable in Practice

**Code concern:** All rate limiter IP extractions trusted `X-Forwarded-For` as the primary IP source. A client sending `X-Forwarded-For: fake-ip` could potentially get its own rate limit bucket per fake IP.

**Empirical test (from server, 7 requests with different fake IPs):**
```
attempt 1 (fake IP 10.0.0.1): Invalid username or password
attempt 2 (fake IP 10.0.0.2): Invalid username or password
attempt 3 (fake IP 10.0.0.3): Invalid username or password
attempt 4 (fake IP 10.0.0.4): Invalid username or password
attempt 5 (fake IP 10.0.0.5): Invalid username or password
attempt 6 (fake IP 10.0.0.6): Too many attempts
attempt 7 (fake IP 10.0.0.7): Too many attempts
```

**Analysis:** All 7 requests hit the SAME rate limit bucket (`127.0.0.1`). This confirms Caddy's default behavior when `trusted_proxies` is not configured: client-supplied X-Forwarded-For headers are NOT appended to — Caddy replaces with the real client IP. The bypass does not work through this Caddy config.

**Hardening applied anyway (defense-in-depth):**
- Caddyfile: `header_up X-Real-IP {remote_host}` added to reverse_proxy block — explicitly sets a trusted IP header that cannot be client-spoofed
- `server.js`: All 6 IP extraction sites updated to prefer `x-real-ip` first: `req.headers['x-real-ip'] || req.headers['x-forwarded-for']?.split(',')[0].trim() || req.socket.remoteAddress`
- Commit: `5fb5a53`
